### 1️⃣ Project Overview

*   **Problem Statement:** How can we design and build systems where multiple AI agents collaborate to solve a problem that is too complex for a single agent? This project explores how to create a structured, multi-step workflow for AI agents using LangGraph, enabling them to work together in a predictable and controllable way.
*   **Target Users:** This project is for developers interested in the emerging field of AI agents and agentic workflows. The target user is familiar with basic LLM concepts and wants to learn how to orchestrate multiple agents into a coherent system that can perform tasks like research and writing.
*   **High-Level System Description:** We will build a multi-agent system for writing a blog post. The system will consist of two agents: a "Researcher" and a "Writer". The workflow is defined as a graph using LangGraph, where each agent is a node. The state (the topic, research notes, and final article) flows from the Researcher node to the Writer node, ensuring a logical and orderly process.

### 2️⃣ Development Model

*   **Chosen Model (Phased):** We will use a phased approach. Agentic systems can get complex quickly, so we'll start with a simple, linear workflow and then introduce more advanced, dynamic features. This allows us to build and test each component of the system in isolation.
*   **Phases of Development:**
    *   **v0.1 (Core Workflow):** The first version is a two-step, linear graph: `Researcher` -> `Writer` -> `END`. This is the core of our system and ensures that the two agents can pass information to each other correctly via the shared state object.
    *   **v0.2 (Adding a Tool):** We will enhance the `Researcher` node by giving it a tool, such as a web search capability. This makes the agent more powerful, as it can now gather real, up-to-date information from the internet instead of just relying on the knowledge baked into the LLM.
    *   **v1.0 (Conditional Workflow):** The final version will introduce a conditional edge. After the `Researcher` node runs, a new `gatekeeper` node will check if the research is good enough. If it is, the graph proceeds to the `Writer`. If not, it loops back to the `Researcher` to try again. This makes the system more robust and autonomous.

### 3️⃣ System Architecture

LangGraph allows us to define our multi-agent system as a **state machine**. The architecture is composed of a central state, a set of nodes that modify the state, and a graph that defines the flow of control between the nodes.

*   **Major Components:**
    *   **The State (`AgentState`):** This is a shared dictionary that acts as the "memory" of the system. All agents can read from and write to this central state object. In our case, it holds the `topic`, the `research_notes`, and the `final_output`.
    *   **The Nodes (`researcher_node`, `writer_node`):** Each node is a Python function that represents an agent or a tool.
        *   **`researcher_node`:** Its responsibility is to take the `topic` from the state, perform research (in our case, by calling an LLM), and write the results back to the `research_notes` field in the state.
        *   **`writer_node`:** Its responsibility is to take the `research_notes` from the state, use them to generate a blog post, and write the result to the `final_output` field in the state.
    *   **The Graph (`StateGraph`):** This is the orchestrator. It defines the structure of the workflow. We define the nodes and then the **edges** that connect them, telling the system which node to run after another.

*   **Data & Control Flow:**

    The control flow is explicitly defined by the graph structure. The state is passed to each node, which in turn returns a dictionary of the fields it has modified. LangGraph automatically updates the central state with these changes.

    ```
    [START]
       │
       ▼
    (State: {topic: "AI", research_notes: "", final_output: ""})
       │
       ├─> Run [Researcher Node]
       │
       ▼
    (State: {topic: "AI", research_notes: "AI is...", final_output: ""})
       │
       ├─> Run [Writer Node]
       │
       ▼
    (State: {topic: "AI", research_notes: "AI is...", final_output: "Blog post..."})
       │
       ▼
    [END]
    ```

## 28. Build a Multi-Agent System With LangGraph (Python)

### Description
This project introduces you to building complex, stateful multi-agent workflows using LangGraph. LangGraph allows you to define agentic workflows as a state graph, where each node in the graph represents an agent or a tool, and the edges represent the transitions between them. This is a powerful paradigm for creating robust and steerable agentic systems that can handle tasks requiring multiple steps and collaboration.

### Prerequisites
LangChain + LangGraph, state machines, local LLM usage, modular pipeline thinking.

### Resources
LangGraph; Ollama; a search tool; shared state schema (TypedDict-like).

### 5️⃣ Step-by-Step Implementation Guide

Let's build our multi-agent system. Open `langgraph_system.py` and follow along as we break down each part of the code.

#### Step 1: Set Up Your Environment

First, we need to install the required libraries and set up the local LLM.

*   **Install Libraries:** In your terminal, run `pip install -r requirements.txt`. This will install `langchain`, `langgraph`, `ollama`, and other necessary packages.
*   **Run Ollama:** This project requires a locally running LLM through Ollama. Make sure the Ollama server is running and you have pulled a model (e.g., `ollama run llama2`).
*   **What it does:** The script begins by trying to connect to Ollama. If it fails, it prints an error and skips the main execution. This is a crucial check to ensure the system has the resources it needs.

#### Step 2: Define the Shared State

The `AgentState` is the central nervous system of our application.

*   **Code:**
    ```python
    class AgentState(TypedDict):
        topic: str
        research_notes: str
        final_output: str
    ```
*   **What it does:** This `TypedDict` defines the schema for our shared state. It's a dictionary with specific keys and value types.
*   **Why it exists:** By defining a clear schema, we make it easy to see what data is being passed between agents. It helps prevent bugs and makes the code easier to understand. Every node in our graph will receive this state object as an input.

#### Step 3: Implement the "Researcher" Agent Node

This is our first agent. Its job is to gather information.

*   **Code:**
    ```python
    def researcher_node(state: AgentState):
        print("--- Researcher Node ---")
        prompt = PromptTemplate.from_template("Give me a brief overview of the topic: {topic}")
        chain = prompt | llm | StrOutputParser()
        research_notes = chain.invoke({"topic": state['topic']})
        print(f"Research Notes: {research_notes[:100]}...")
        return {"research_notes": research_notes}
    ```
*   **What it does:**
    1.  It takes the current `state` as input.
    2.  It uses a simple LangChain Expression Language (LCEL) chain to ask the LLM for an overview of the `topic` from the state.
    3.  It returns a dictionary with the new `research_notes`. LangGraph will automatically update the central state with this new information.

#### Step 4: Implement the "Writer" Agent Node

This agent's job is to use the researcher's work to create the final product.

*   **Code:**
    ```python
    def writer_node(state: AgentState):
        print("--- Writer Node ---")
        prompt = PromptTemplate.from_template(
            "Based on these notes, write a short blog post:\n\n{research_notes}"
        )
        chain = prompt | llm | StrOutputParser()
        final_output = chain.invoke({"research_notes": state['research_notes']})
        print("Blog post generated.")
        return {"final_output": final_output}
    ```
*   **What it does:**
    1.  It takes the current `state` (which now includes the research notes).
    2.  It uses another LCEL chain to ask the LLM to write a blog post based on the `research_notes`.
    3.  It returns a dictionary with the `final_output`.

#### Step 5: Build and Compile the Graph

This is where we define the workflow and connect the agents.

*   **Code:**
    ```python
    workflow = StateGraph(AgentState)
    workflow.add_node("researcher", researcher_node)
    workflow.add_node("writer", writer_node)

    workflow.set_entry_point("researcher")
    workflow.add_edge("researcher", "writer")
    workflow.add_edge("writer", END)

    app = workflow.compile()
    ```
*   **What it does:**
    1.  `workflow = StateGraph(AgentState)`: We initialize the graph, telling it to use our `AgentState` schema.
    2.  `workflow.add_node(...)`: We add our two agent functions as nodes, giving each a unique string name.
    3.  `workflow.set_entry_point("researcher")`: We declare that the `researcher` node is the first step in the workflow.
    4.  `workflow.add_edge("researcher", "writer")`: We create a directed edge. After the `researcher` node finishes, the `writer` node should run next.
    5.  `workflow.add_edge("writer", END)`: After the `writer` node finishes, the workflow ends.
    6.  `app = workflow.compile()`: This compiles our graph definition into a runnable LangChain object.

#### Step 6: Run the System

Finally, we invoke the compiled graph with our initial input.

*   **Code:**
    ```python
    initial_state = {"topic": "The latest advancements in AI"}
    final_state = app.invoke(initial_state)
    print(final_state['final_output'])
    ```
*   **What it does:**
    *   We create the initial state, providing the starting `topic`.
    *   `app.invoke(initial_state)`: This kicks off the entire workflow. LangGraph handles passing the state between the nodes according to the edges we defined.
    *   The final state, containing the completed blog post, is returned and printed.

### 7️⃣ Debugging & Failure Guide

Agentic systems can be complex to debug. Here are some common problems you might encounter.

*   **Symptom:** The script prints "Failed to connect to Ollama" and then "Skipping LangGraph execution..."
    *   **Problem:** Your Python script cannot communicate with the local Ollama LLM server.
    *   **How to Debug:**
        1.  **Is Ollama running?** Make sure the Ollama application is open. On some systems, you may need to run the command `ollama serve` in a separate terminal.
        2.  **Have you pulled a model?** You need to have at least one LLM downloaded. Run `ollama list` to see your models, or `ollama run llama2` to download and run the default model.
        3.  **Firewall or Network Issues:** Ensure that no firewall is blocking the connection to Ollama, which typically runs on `http://localhost:11434`.

*   **Symptom:** A node runs, but the next node doesn't seem to get the data. You see an error like `KeyError: 'research_notes'`.
    *   **Problem:** The node that was supposed to add data to the state didn't return it in the correct format.
    *   **How to Debug:**
        1.  **Check the `return` statement:** The function for each node **must** return a dictionary where the keys match the keys in the `AgentState` `TypedDict`. For example, `researcher_node` must `return {"research_notes": notes}`. If it just returns `notes`, LangGraph won't know which field in the central state to update.
        2.  **Check for Typos:** Ensure the key in your return dictionary (`"research_notes"`) exactly matches the key in the `AgentState` definition.

*   **Symptom:** The graph fails to compile or run, with a validation error about cycles or missing entry points.
    *   **Problem:** The structure of your graph is invalid.
    *   **How to Debug:**
        1.  **Missing Entry Point:** Make sure you have `workflow.set_entry_point("your_first_node")`. Every graph needs a beginning.
        2.  **Incomplete Path:** Every node must have a path that eventually leads to `END`. If you add a node but don't add an edge leading from it to another node (or to `END`), the graph is a dead end.
        3.  **Check Your Edges:** Double-check the string names in your `workflow.add_edge("source_node", "destination_node")` calls. A typo here is a common cause of errors.

### 1️⃣1️⃣ Production Thinking (Reflection Only)

This simple two-agent system is a great start. What would happen if we deployed it as a real application?

*   **What would break at scale?**
    *   **Infinite Loops:** Our v1.0 design includes a conditional loop (`Researcher` -> `Gatekeeper` -> `Researcher`). What happens if the researcher is never able to produce "good enough" research? The system could get stuck in this loop forever, consuming resources and never producing an output. A production system would need a "max retries" counter in the state to break out of the loop after a certain number of attempts.
    *   **LLM Rate Limits:** Our system makes multiple calls to an LLM. If we were using a cloud API like OpenAI, and we had many users running this workflow at once, we could easily hit the API's rate limits, causing our application to fail. A production system needs robust error handling, backoff-and-retry logic, and potentially a load balancer to distribute calls across multiple API keys.

*   **What would need monitoring?**
    *   **Cost per Run:** Every LLM call costs money. We would need to track the number of LLM calls per workflow invocation to understand and control our costs. If a workflow gets stuck in a loop, costs could spiral out of control.
    *   **Latency per Node:** We would need to monitor the time it takes for each node to run. If the `researcher_node` suddenly becomes very slow, it might indicate a problem with our tool (e.g., the web search API is down).
    *   **Graph Trajectory:** We would want to log the path that each workflow takes through the graph. How often does the `gatekeeper` send the workflow back to the researcher? This information is invaluable for understanding and improving the system's behavior.

*   **One Security Concern:**
    *   **Tool-Related Prompt Injection:** Giving an agent a tool, especially a web search tool, opens up a new attack vector. A malicious user could craft a topic that tricks the researcher into searching for and retrieving harmful content from the web. This content could then be passed to the writer agent and potentially included in the final output. Production systems need strict validation and sanitization of any data retrieved from external tools.
